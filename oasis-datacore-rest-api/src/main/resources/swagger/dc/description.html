Allows to manage (CRUD) and find Data Resources and their Models.

  - Data Resources are handled in JSON-LD-like (implicit context) format (see also playground) and with W3C LDP (Linked Data Platform, see primer and wiki)-like operations (URIs, future collection filtering (1, 2) -inspired finders etc.). Have a look at existing Resources, at /dc/type/!pli:city for instance for resources in the !pli:city Model type. In addition to the simpler default native pure JSON format, "true" JSON-LD formats and semantic web formats (such as RDF, see example) are available.

  - Data Models describe what kinds of Data Resources are allowed in a JSON Schema-like format (see jsonschema.net playground) with string, boolean, int, float, long, double, date, map, list, i18n, resource fields grouped in reusable Mixin types. Have a look at known Models at /dc/type/dcmo:model and Mixins at /dc/type/dcmi:mixin (beta, see full Model definitions at /dc/model), in their own metamodel where they may be queried as Resources. This allows to know in which Model types a given kind of Resources can be queried, for instance :
      - dcmo:globalMixins=!pl:place : lists Models that describe a location i.e. have a pl:shape WKS point, such as : !co:company, whose WKS can then be retrieved, for instance by GET /dc/type/!co:company .
      - dcmo:globalFields.dcmf:name=!plo:country_name : lists Models that "officially" have a country (i.e. having the "official" field for a country in OASIS Datacore), and that can therefore be queried on that, such as : !plo:country, !pli:city, !co:company, !cityareauseit:urbanAreaDestinationOfUse, !cityarea:cityArea , which can then be queried by country, for instance by GET /dc/type/!plo:country, GET /dc/type/!pli:city, GET /dc/type/!co:company, GET /dc/type/!cityareauseit:urbanAreaDestinationOfUse and GET /dc/type/!cityarea:cityArea.

See wiki for further examples, FAQ, how to & common use cases, cookbook (writing clients...) and full documentation.


This endpoint has for base URL ${datacoreApiServer.baseUrl} (URL that must be called by clients using HTTP) and for container URL ${datacoreApiServer.containerUrl} (URL that must be used to build Datacore Data resource URIs).

URIs of Datacore Data resource are in the form $containerUrl/dc/type/$type/$iri, where $containerUrl is unique for a container (ex. http://data.oasis-eu.org), $type is among available model types and $iri is the resource's Internal Resource Identifier (IRI). IRIs have 3 constraints : they must be valid URL endings, unique (not enforced yet), and if possible readable and representative of the resource.

Using operations where type of data is provided in URL usually improves performances : '*inType' methods are faster, finder queries with type even more.

WARNING Swagger UI has some limitations (but not other HTTP clients) :
  - it encodes path parameters, so type & IRI will be wrong if they have encodable characters (ex. slash in IRI).
  - it weirdly encodes query parameter values, notably '+' as space (which is especially annoying when specifying ASC order in Datacore queries). It probably has to do with application/x-www-form-encoded encoding.

Authentication:
  - In production (and integration), OASIS Kernel OAuth2 authentication must be used, i.e. HTTP requests' Authorization header must be "Bearer <base 64-encoded valid OAuth2 Bearer / Access Token>" (see how to get one).
  - In dev mode, there is a mock authentication system that supports Basic Auth (i.e. HTTP requests' Authorization header must be "Basic <base 64-encoding of username:password>"), for a few users without password, see available ones in configuration.
  - Datacore API's Swagger UI allows to provide said Authorization header, with a default value that logs as admin in dev mode.
